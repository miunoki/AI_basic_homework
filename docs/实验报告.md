## 摘要

本项目围绕"校园智能问答助手"任务，基于浙江大学 CC98 论坛真实帖子构建校园生活知识库，并采用 RAG（Retrieval-Augmented Generation，检索增强生成）架构接入 DeepSeek 大语言模型，实现面向新生入学与校园生活场景的智能问答系统。系统从 CC98 论坛 9 个校园相关版块采集约 3640 条高质量帖子，运行时实际加载 3643 个知识条目；用户提问后，系统先通过检索模块召回相关帖子，再将参考资料与问题拼接为 Prompt 交给 DeepSeek 生成回答，并在回答末尾标注参考来源。

在基础功能方面，系统完成了知识库构建、检索增强问答、多轮对话、边界问题处理和命令行交互；在进阶功能方面，进一步实现了来源标注、Gradio 网页界面、语义向量检索方案、RAG 与纯大模型对比实验以及安全配置规范。运行截图表明，系统能够准确回答"图书馆怎么预约"等校园问题，能在"苹果手机和华为哪个好"等知识库外问题上拒绝编造，并能结合上一轮对话回答"预约后多久签到"等连续追问。

**关键词**：RAG；校园问答；DeepSeek；CC98；检索增强生成；Gradio；Prompt 工程

## 一、项目概述

### 1.1 选题背景

新生入学阶段会遇到大量具体而琐碎的问题，例如图书馆座位如何预约、校医院牙科体验如何、紫金港附近有什么好吃的、宿舍条件如何、校园网和 VPN 如何使用等。官方通知通常更规范，但不一定覆盖学生真实生活中的细节；论坛经验帖则更接地气，但分散在不同版块，人工查找成本较高。

通用大模型对学校内部论坛、校园设施和学生经验缺乏实时、可靠的知识，直接提问容易出现幻觉。本项目使用 RAG 技术，把真实论坛帖子作为知识来源，让大模型"先查资料，再回答"，从而提升回答的准确性、可解释性和可追溯性。

### 1.2 项目目标

- 构建一个基于浙江大学 CC98 论坛真实帖子的新生入学指南知识库
- 实现面向校园生活问题的 RAG 问答流程：用户问题 → 检索知识库 → 拼接 Prompt → 调用 DeepSeek → 生成回答
- 支持上下文连续追问，避免"一问一答"式的上下文丢失
- 对资料库之外的问题如实说明资料不足，不凭空编造
- 提供命令行和 Gradio 网页两种交互界面，便于演示和使用

## 二、总体方案设计

### 2.1 系统架构

系统采用典型 RAG 架构，核心流程如下：

```
用户输入问题
       ↓
检索器根据问题从 knowledge/ 中召回 Top-K 相关帖子
       ↓
build_prompt() 将参考帖子、用户问题和来源信息拼接为 Prompt
       ↓
llm.py 调用 DeepSeek API 生成自然语言回答
       ↓
主程序输出回答，并记录历史对话与参考来源
```

| 模块 | 文件 | 功能说明 |
|------|------|---------|
| 知识库 | knowledge/*.txt | 保存 CC98 帖子的版块、标题、正文和精选回复 |
| 检索模块 | retriever.py | 加载知识库，进行关键词检索或语义向量检索 |
| 模型封装 | llm.py | 封装 DeepSeek API，统一处理消息和编码异常 |
| 命令行入口 | main.py | 实现多轮对话、Prompt 构建、来源标注 |
| 网页界面 | app.py | 基于 Gradio 的可视化聊天页面 |
| 对比实验 | compare.py | 测试 RAG 与纯大模型回答质量差异 |
| 配置文件 | config.example.py | 提供 API Key 配置模板，避免密钥硬编码 |

### 2.2 数据流与问答流程

1. 用户在命令行或网页界面输入问题
2. 系统读取当前问题，并在短追问场景下拼接上一轮问题作为检索查询
3. 检索器从 3643 个知识条目中返回最相关的 2-3 个帖子
4. Prompt 构建函数截断过长帖子，保留标题、正文和精选回复，避免超过上下文长度
5. DeepSeek 根据参考帖子生成回答，并在回答末尾标注引用来源
6. 系统将原始用户问题和回答保存到 history 中，用于后续多轮对话

### 3.2 爬虫与清洗策略

爬虫通过 CC98 REST API 分页获取帖子列表，每页 20 条，并对正文和回复进行 BBCode 清洗。为保证质量，系统过滤零回复帖子，保留最多 30 条精选回复，并对文件名中的非法字符进行清理。

```python
def fetch_board_topics(board_id, max_count=400):
    """分页获取版块帖子列表，直到 API 返回空。"""
    all_topics = []
    for offset in range(0, max_count, 20):
        batch = api_get(
            f"/board/{board_id}/topic",
            {"from": offset, "size": 20}
        )
        if not batch or len(batch) < 20:
            break
        all_topics.extend(batch)
        time.sleep(0.15)  # 温和限速，避免对服务器造成压力
    return all_topics
```

## 四、核心技术实现

### 4.1 检索算法：中文 2-gram + Jaccard + 标题加权

原始脚手架使用简单字符重合度，容易使长文本获得更高分，即使内容并不相关。本项目将检索模块升级为中文 2-gram 分词、Jaccard 相似度和标题加权。标题通常是帖子内容的高密度摘要，因此标题命中被赋予更高权重。

```python
def _tokenize(text):
    text = re.sub(r'[^一-龿\w]', ' ', text)
    words = [w for w in text.split() if len(w) >= 1]
    bigrams = []
    for w in words:
        if len(w) >= 2:
            bigrams.extend(w[i:i+2] for i in range(len(w)-1))
    return set(words + bigrams)

def score(chunk):
    title_hits = len(q_tokens & title_tokens)
    jaccard = len(q_tokens & body_tokens) / len(q_tokens | body_tokens)
    phrase_bonus = sum(1 for t in q_tokens if len(t) >= 2 and t in text)
    return title_hits * 10 + jaccard * 3 + phrase_bonus * 0.5
```

| 版本 | 核心方法 | 主要问题/效果 |
|------|---------|-------------|
| v1 | 字符重合度 | 长文档容易"吃掉"分数，检索不稳定 |
| v2 | 关键词匹配 + 标题加权 | 通用词仍有干扰 |
| v3 | 2-gram + Jaccard + 标题加权 | 对图书馆、校医院、美食等问题检索更准确 |

### 4.2 Prompt 工程与幻觉抑制

系统提示词将模型定位为"浙江大学校园生活助手"，并明确要求"只根据参考帖子回答，完全不相关时说明资料库中暂未找到相关信息"。这种提示词设计将回答风格、知识边界和拒答策略同时写入模型行为约束中。

```python
SYSTEM_PROMPT = """你是浙江大学「校园生活助手」，专门为新生解答校园生活中的各种问题。
你的知识全部来自 CC98 论坛上浙大学生们的真实讨论和经验分享。

回答规则：
1. 认真阅读【参考帖子】中的内容，提取有用信息来回答
2. 如果帖子里确实有相关信息，就直接引用回答，语气自然友好
3. 只有当所有参考帖子都和问题完全不相关时，才说「抱歉，资料库中暂未找到相关信息」
4. 回答要简洁、口语化，像一个热心的学长/学姐
5. 回答末尾用「📎 参考来源：」标注你依据了哪些帖子的标题
"""
```

### 4.3 DeepSeek API 封装

`llm.py` 使用 OpenAI 兼容接口调用 DeepSeek，并支持从 `config.py` 或环境变量读取 API Key。为了避免提交密钥，`config.py` 被 `.gitignore` 排除，只提交 `config.example.py` 作为模板。

```python
from openai import OpenAI

client = OpenAI(api_key=DEEPSEEK_API_KEY, base_url="https://api.deepseek.com")

def chat(messages, model="deepseek-chat", temperature=0.7):
    resp = client.chat.completions.create(
        model=model, messages=messages, temperature=temperature
    )
    return resp.choices[0].message.content
```

### 4.4 多轮对话与上下文增强检索

多轮对话不仅需要把历史消息发给模型，还需要解决短追问的检索问题。例如用户先问"图书馆怎么预约？"，再问"预约后多久签到？"，如果只用第二句检索，可能检索到"演出预约"等不相关帖子。因此主程序在短问题场景下将上一轮问题拼接进检索查询，提升连续追问命中率。

```python
search_query = question
if len(history) >= 3:
    last_question = history[-2]["content"]
    if len(question) <= 15:
        search_query = last_question + " " + question

related = retrieve(search_query, chunks, mode=RETRIEVAL_MODE)
```

### 6.2 RAG 与纯大模型对比

`compare.py` 选取 12 个校园生活问题，分别测试 RAG 模式和无知识库模式。评价维度包括是否有效回答、是否拒绝编造、是否可能幻觉。由于校园问题依赖本校内部经验，纯大模型容易给出泛化或编造性回答，而 RAG 模式能依托真实帖子给出更接地气的答案。

| 指标 | RAG（带知识库） | 无知识库 |
|------|:---:|:---:|
| 有效回答比例 | 约 83% | 约 42% |
| 未回答/拒绝编造 | 约 17% | 约 0% |
| 可能幻觉 | 约 0% | 约 58% |
| 平均耗时 | 3-5 秒 | 1-3 秒 |
| 可追溯性 | 有参考来源 | 无参考来源 |

### 6.3 性能分析

| 环节 | 耗时/规模 | 说明 |
|------|----------|------|

### 6.4 失败案例与改进方向

当前关键词检索对"图书馆""校医院""紫金港"等明确词汇表现较好，但对"怎么算""注意什么"等通用表达仍可能受到噪声影响。后续可以加入停用词表、版块过滤、BM25 排序或语义向量检索，以进一步提升召回质量。

### 7.2 分工与 AI 协作方式

小组三人均独立使用 CC+DS 完成各自负责的模块，具体分工见第八节。AI 负责代码生成和 bug 定位，每位成员对自己模块的代码进行阅读理解、功能验证和必要的人工修改——确保答辩时能逐行讲解。

### 7.3 遇到的坑

- **CC98 Token 过期**：爬虫中途 401，需手动从浏览器 localStorage 提取新 Token，AI 无法自动完成
- **翻页 bug**：AI 生成的初始爬虫用 `len(batch) < 30` 判断翻页结束，但 API 每页最多 20 条导致只爬一页——人工读代码发现并改为 `< 20`
- **编码差异**：openai 库的 surrogate 字符报错在不同终端表现不一致，AI 给的修复方案需人工在不同环境验证
- **多轮对话记忆丢失**：CC+DS 生成的上下文管理代码在追问时丢失历史，换用 ChatGPT 定位根因并修复
- **AI 偶尔编造参数**：给过不存在的 Gradio API 参数名，实测报错后人工查文档纠正

### 7.4 体会

1. AI 在代码生成、bug 定位、文档整理方面效率极高，但生成的代码必须逐行读、实际跑、验证结果
2. 不同 AI 工具有不同强项——CC+DS 主力开发，ChatGPT 在特定 bug 修复上更精准
3. 人工不可替代的环节：需求判断、方案取舍、检索效果验证、安全审查
4. 「AI 写代码，人来定方向、审逻辑、做决策」——方向盘始终在自己手里
5. 答辩时每位成员都能讲清楚自己模块的代码逻辑，不存在「AI 写的我看不懂」的情况
